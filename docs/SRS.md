# 1. 引言

<a id="11-编写目的"></a>
## 1.1 编写目的

本文档为「SmartDine（智餐 AI）· 多商户餐厅一体化运营管理平台」的软件需求规格说明书（SRS），用于定义系统的功能需求与非功能需求，明确四个 AI 智能体（点餐推荐、库存预测、客服处理、经营分析）的深度业务逻辑与基于 Dify 工作流的实现方案，并界定**平台管理端、商户管理端与用户端**的需求范围，为后续的概要设计、详细设计、开发实现与测试验收提供统一依据。

相比 V1.0，V2.0 的核心升级方向为：

1. **多商户 SaaS 化**：从单店系统升级为支持多商户入驻的平台型产品，每个商户拥有独立的数据空间与管理后台。
2. **AI 智能体深度扩展**：四个智能体从基础功能升级为具备多轮推理、多源数据融合、预测建模、策略优化等高级能力的复杂工作流。
3. **点餐推荐引擎重磅打造**：从简单的「规则+LLM推荐」升级为**混合推荐引擎**（协同过滤+内容推荐+知识图谱+深度学习），支持多模态交互、群体点餐智能、营养健康管理、实时行为感知等前沿能力，作为整个项目的**技术亮点与答辩核心**。
4. **智能体必要性设计（V2.0 核心设计原则）**：每个 AI 智能体在业务流程中具有**明确的存在刚性**——点餐推荐智能体虽与传统菜单并行，但提供个性化推荐、过敏原安全检测、群体点餐、智能凑单、营养管理等传统方式无法做到的差异化能力；库存预测复杂度远超人工计算能力；客服需同时处理远超人力上限的并发会话；经营分析需融合多维度数据发现人工无法察觉的隐性模式。**智能体不是可选插件——三大智能体（库存/客服/经营）为刚性依赖，点餐推荐为核心差异化竞争力。**

本文档面向的读者包括项目组开发与设计成员、指导教师以及测试与验收人员。文档中标注为「建议」的参数与阈值可结合实际情况调整，但功能边界、技术选型与智能体的输入输出契约应作为开发基线保持一致。

<a id="12-项目背景"></a>
## 1.2 项目背景

传统餐厅在点餐、库存、客服与经营决策等环节高度依赖人工经验，存在推荐转化低、食材积压或缺货、客诉响应慢、数据难以沉淀为决策等痛点。此外，现有餐饮SaaS产品大多面向单店，中小型连锁或美食广场等「多档口/多门店」场景缺乏统一管理和跨店智能协同能力。

本项目在 V1.0 单店版本的基础上，升级为一套「**多商户 SaaS + AI 智能体**」融合的运营平台：

- **平台层**：支持多个独立商户（不同餐厅品牌/门店）入驻，平台超级管理员统一管理商户入驻审核与平台运营。
- **商户层**：每个商户拥有独立的管理端控制台，管理自己的菜品、订单、桌台、库存、会员、员工及智能体配置，各商户数据完全隔离。
- **顾客层**：用户端微信小程序支持顾客浏览附近多家商户、选择餐厅、扫码点餐，实现「一个入口，多家选择」的便捷体验。
- **智能层**：四个 AI 智能体作为平台的智能中枢，分别在点餐、备货、客服、分析四条业务线上提供深度自动化与辅助决策能力，且智能体可基于商户级数据进行个性化适配。

本系统面向平台运营方提供平台管理端，面向各商户管理者提供商户管理端 Web 控制台，面向到店顾客提供微信小程序用户端。四个 AI 智能体是系统的核心差异化能力，基于 Dify 工作流深度编排实现，使经营各环节具备自动化与辅助决策能力。

# 2. 项目总体描述

<a id="21-产品定位与建设目标"></a>
## 2.1 产品定位与建设目标

**产品定位：** SmartDine（智餐 AI）是面向多商户的「**AI 原生（AI-Native）**」餐饮 SaaS 运营管理平台。**「AI 原生」的含义是：智能体是产品的核心差异化竞争力，而非可有可无的附加功能。** 库存预测、客服处理、经营分析三个智能体为刚性依赖——离开了它们，相应的多因子备货计算、高峰期并发客服、跨维度经营分析将无法进行；点餐推荐智能体与传统菜单通道并存，但在个性化推荐、过敏原安全检测、群体点餐、智能凑单、营养管理等能力上提供了传统方式不可企及的价值。以「一个平台、多方受益」为设计理念——平台方负责商户管理与平台运营，各商户独立管理自己的餐厅业务，顾客在一个小程序中即可浏览和选择多家餐厅。

**总体建设目标：**

- **智能化点餐（核心亮点）：** 构建**混合推荐引擎**，融合协同过滤、内容推荐、知识图谱、深度学习 CTR 预估与 LLM 语义理解，支持多模态交互（文本+语音+图片），在群体点餐、营养健康管理、动态定价感知、实时行为适应等维度做到行业前沿水平。**这是项目的技术制高点与答辩重点展示模块。系统同时保留传统菜单手动加购通道作为兜底，但 AI 对话通道在个性化推荐、过敏原安全检测、群体点餐、智能凑单、营养管理等能力上具有压倒性优势——智能体不是「可选的推荐帮手」，而是产品的核心竞争力所在。**
- **精准化备货：** 基于多维度时序预测（趋势+周期+节假日+天气+促销因子），结合供应商可靠性评分与动态安全库存计算，自动生成可执行的采购计划。**食材数量多（50+）、预测因子复杂，人工根本无法在合理时间内完成同等级别的计算——智能体是唯一可行的备货决策方式。**
- **自动化客服：** 多轮对话式客服具备意图深层理解、情绪变化追踪、流失风险预警、知识库自增长、主动外呼通知等高级能力。**高峰时段同时涌入数十条咨询，人工客服（1–2人）不可能即时响应——智能体是唯一能保证响应速度的方案。**
- **数据化决策：** 具备异常检测、归因分析、What-if 模拟、菜单工程分析、顾客终身价值建模等能力。**跨维度（订单×评价×库存×客流×营销）的关联分析超出人脑处理能力——智能体是唯一能发现隐性经营问题的手段。**
- **多商户可运营：** 构建完整的商户入驻、审核、权限分配、功能订阅、数据隔离体系，使系统可作为真实 SaaS 产品运营。

<a id="22-系统组成与总体架构"></a>
## 2.2 系统组成与总体架构

系统由四个逻辑层构成，自上而下为：

| **层次** | **组成与职责** |
|---|---|
| 表现层 | **平台管理端**（Web 控制台，平台超级管理员使用，管理商户入驻与平台运营）；**商户管理端**（Web 控制台，各商户管理者使用，管理自家餐厅）；**用户端**（微信小程序，顾客使用，含选店、点餐、客服、会员等）。 |
| 业务服务层 | 提供商户管理、菜品、订单、桌台、营销、库存、会员、员工、权限等业务能力的后端服务，统一对外提供 REST API。所有接口基于商户上下文进行数据隔离，并作为智能体读写业务数据的网关。 |
| 智能层 | Dify 平台（本地虚拟机自托管）承载四个智能体的 Chatflow / Workflow / Agent，编排意图识别、知识检索（RAG）、模型推理、代码计算与工具调用。每个智能体内部包含多个子工作流形成复杂编排。业务服务层通过 Dify API 调用智能体。 |
| 数据层 | 关系型数据库（MySQL 8.0）存储结构化业务数据，以 `merchant_id` 作为隔离键；Dify 知识库按商户维度组织菜单、FAQ 等非结构化语料；Redis 用于会话缓存与热点数据加速；并对接支付、短信、大模型等第三方服务。 |

**关键设计原则：**

- **AI 原生（智能体为核心差异）：** 智能体是产品的核心竞争力。点餐推荐智能体与传统菜单并行存在，但提供了传统方式无法匹敌的个性化推荐、安全检测与高级功能（群体点餐/营养管理/智能凑单）；库存预测、客服处理、经营分析三个智能体则为刚性依赖——其解决的问题（多因子计算/并发响应/跨维度分析）人工无法在合理时间内完成。系统不提供这三个智能体的手动替代方案。
- **商户数据隔离：** 所有业务数据通过 `merchant_id` 进行租户隔离，查询默认带租户条件，防止数据跨商户泄露。
- **业务与智能体解耦：** 智能体不直接操作数据库，而是通过业务服务层提供的接口读取上下文并回写结果。
- **平台—商户权限分层：** 平台级操作（商户入驻审核、平台配置）与商户级操作（菜品管理、订单处理）严格分离。
- **智能体商户适配：** 同一套 Dify 工作流，通过传入不同的商户上下文（菜单、历史数据、偏好模型），为每个商户提供个性化智能服务。

<a id="23-多商户体系设计"></a>
## 2.3 多商户体系设计

### 2.3.1 商户模型

系统采用「平台—商户—用户」三层组织模型：

```
平台（Platform）
├── 商户 A（Merchant A）- 川味面馆
│   ├── 管理员（店长）
│   ├── 员工（收银/后厨/服务员）
│   ├── 菜品 / 订单 / 库存 / 会员 ...
│   └── AI 智能体配置
├── 商户 B（Merchant B）- 粤式茶餐厅
│   ├── 管理员（店长）
│   ├── 员工
│   ├── 菜品 / 订单 / 库存 / 会员 ...
│   └── AI 智能体配置
└── 商户 C（Merchant C）- 日式料理
    └── ...
```

### 2.3.2 商户入驻流程

```
商户注册 → 填写商户资料 → 提交审核 → 平台审核（通过/驳回）→ 开通商户空间 → 初始化数据 → 商户登录管理
```

- **注册与资料提交：** 商户填写餐厅名称、地址、联系电话、营业执照（可选）、经营品类、桌台数量等基本信息。
- **平台审核：** 平台超级管理员审核商户资料，通过后自动为该商户创建独立数据空间与默认角色。
- **初始化：** 系统为新商户自动创建默认菜品分类、桌台模板，并初始化智能体默认配置。
- **开通状态：** 商户状态包括：待审核 / 已开通 / 已停用 / 已注销。

### 2.3.3 数据隔离策略

| **隔离维度** | **策略** |
|---|---|
| 数据库层面 | 所有核心业务表包含 `merchant_id` 列，作为租户隔离键。业务查询统一通过 ORM 层自动注入 `WHERE merchant_id = ?` 条件。 |
| 缓存层面 | Redis key 以 `{merchant_id}:` 为前缀，各商户缓存空间隔离。 |
| 文件存储 | 菜品图片等静态文件按 `/{merchant_id}/` 路径组织，隔离商户间文件访问。 |
| 知识库 | Dify 知识库按商户创建独立知识库或数据集，RAG 检索时限定在对应商户范围内。 |
| API 鉴权 | 后端从 JWT Token 解析 `merchant_id`，所有 API 在网关层校验租户归属，非该商户数据拒绝访问。 |

<a id="24-用户角色与权限"></a>
## 2.4 用户角色与权限

| **角色** | **使用端** | **主要权限与职责** |

| 商户管理员（店长） | 商户管理端 | 所属商户的全部模块查看与管理权限；菜品/营销上下架、采购确认、客服转人工处理、员工与权限配置、智能体参数配置。 |

| 系统（自动账户） | — | 代表智能体或定时任务执行的自动操作（如出库扣减、自动回复、生成摘要），在记录中以「系统」标识，并标注所属商户。 |

## 2.5 技术选型

| **类别** | **技术方案** | **说明** |
|---|---|---|
| 平台管理端前端 | Vue 3 + Vite | 平台超级管理员使用的 Web 控制台。 |
| 商户管理端前端 | Vue 3 + Vite | 各商户管理者使用的 Web 控制台，与平台端共享 UI 组件库，但路由与权限隔离。 |
| 用户端 | 微信小程序 | 面向顾客的扫码点餐入口，新增「商户选择」与「附近商户」功能。 |

| 数据库 | MySQL 8.0 + Redis | MySQL 存储结构化业务数据（全部以 merchant_id 做租户隔离）；Redis 用于会话管理与热点数据缓存。 |

| 第三方 | 微信支付、短信服务、微信地图 | 用户端在线支付、会员注册验证码、附近商户距离计算。 |

<a id="26-运行环境"></a>
## 2.6 运行环境

- **平台管理端 & 商户管理端：** 主流桌面浏览器（Chrome / Edge 最新版），推荐 1280px 以上分辨率，并对窄屏做基础适配。
- **用户端：** 微信小程序，运行于 iOS / Android 主流机型的微信客户端，竖屏移动端布局。
- **服务端：** Linux 服务器运行后端服务与 MySQL 数据库；Dify 部署于本地虚拟机（自托管），通过内网与后端服务通信。
- **约束：** 智能体的所有数据读写必须经业务服务层接口，不得绕过业务系统直接访问数据库，以保证数据一致性与可审计性。
- **约束：** 涉及资金（支付、退款）与权限变更（如角色权限）的操作必须有明确的人工确认与操作记录，AI 不得自动执行不可逆的资金操作。
- **约束：** 智能体输出（推荐、采购、回复、摘要）默认作为「建议」呈现，关键动作（确认下单、生成采购订单、发送客服回复）需人工二次确认或可由规则配置为自动。
- **假设：** 初期支持 3–10 个商户入驻，数据规模为中小量级；系统架构天然支持横向扩展至更多商户。
- **假设：** 演示阶段可使用模拟/历史数据驱动智能体，真实接入后逻辑保持一致。

<a id="27-设计约束与假设"></a>
## 2.7 设计约束与假设

- 每个商户的菜单、订单、库存等数据完全隔离，商户间不可相互访问。
- 顾客会员账号在平台层面统一，但积分、等级、优惠券按商户独立管理（即顾客在商户 A 的积分不能用于商户 B）。
- 用户端小程序支持顾客同时浏览多家商户的菜单并分别下单（不同商户的订单独立管理）。
- 智能体的模型调用与知识库按商户区分上下文，保证推荐的个性化与数据安全。

<a id="3-管理端功能需求"></a>
# 3. 管理端功能需求

管理端分为**平台管理端**（超级管理员）与**商户管理端**（各商户管理者）两个层面。本节按级描述。

<a id="30-平台管理端超级管理员"></a>
## 3.0 平台管理端（超级管理员）

平台管理端为本版本新增模块，供平台运营方管理所有入驻商户与平台整体运营。

### 3.0.1 平台运营总览

- **平台 KPI 卡片：** 展示平台累计商户数、活跃商户数、总订单量、总 GMV、待审核商户数。
- **商户分布：** 按品类、地域（如有）展示商户分布图。
- **平台趋势：** 近 30 日新增商户趋势、订单总量趋势。
- **待办事项：** 突出展示待审核商户、被投诉商户等需要平台介入的事项。

### 3.0.2 商户管理

- **商户列表：** 以表格展示全部商户（名称、品类、地址、联系人、联系电话、入驻时间、状态：待审核/已开通/已停用/已注销），支持按状态筛选与按名称搜索。
- **商户审核：** 查看商户提交的入驻资料，执行审核通过（自动创建商户空间）或驳回（附驳回理由）。
- **商户详情：** 查看商户基本信息、当前订阅状态、最近订单量、最近登录时间。
- **状态管理：** 支持停用商户（商户无法登录管理端，用户端不再展示该商户）与重新启用。
- **商户配置：** 为商户设置功能开关（如是否开启智能点餐、最大桌台数限制等）。

### 3.0.3 平台设置

- **平台信息：** 维护平台名称、客服电话、关于我们等。
- **管理员账号：** 管理平台超级管理员的账号与权限。
- **系统参数：** 如默认商户试用期限、商户最大数量等全局参数。

<a id="31-运营总览"></a>
## 3.1 商户管理端——运营总览

作为商户登录后的首页，提供当日经营全貌与智能体协同状态。

- **经营 KPI 卡片：** 展示今日营收（含同比/环比趋势）、今日订单（含客单价）、库存预警项数、待处理客服工单数、推荐采纳率（新增），预警类指标以醒目标签提示。
- **常用操作：** 提供「新增菜品、录入库存、创建活动、查看订单」等快捷入口。
- **智能体协同状态：** 以四张卡片展示各智能体的运行状态与关键产出（推荐采纳数/采纳率、建议采购额与预警项、自动解决率与待处理工单、经营摘要洞察），点击进入各自工作台。
- **经营趋势：** 展示近 7 日营收趋势图、客流时段分布图。
- **智能体动态流：** 实时信息流展示各智能体最近的自动动作及时间（如「12:30 点餐推荐：推荐了酸辣粉给 A5 桌，已采纳」）。
- **门店状态：** 顶部展示营业状态（营业中/休息中）与在线智能体数量。

<a id="32-菜品管理"></a>
## 3.2 商户管理端——菜品管理

维护菜单全部菜品，是点餐推荐与库存联动的数据基础。

- **列表与检索：** 以表格展示菜品（缩略图、名称、主料、分类、售价、标签、本周销量、推荐权重、状态），支持按分类分段筛选与按名称搜索。
- **新增/编辑：** 通过弹窗维护菜品名称、分类、售价、成本价、主要食材、详细描述、标签（招牌/热销/特价/新品/限时/健康轻食/高蛋白/素食）、过敏原标注（花生/麸质/乳制品/海鲜/蛋类/大豆/坚果）、营养信息（热量/蛋白质/碳水/脂肪，可选）、菜品图片、推荐权重（影响推荐排序）与上下架状态。
- **推荐权重配置：** 商户可为每个菜品设置推荐权重（1–10），影响推荐引擎的基础排序；权重可在「菜品管理」页直接调整。
- **上下架与删除：** 支持单菜品上架/停售切换与删除；停售菜品不参与用户端展示与点餐推荐。
- **食材关联（BOM）：** 在菜品维护中关联其所需食材及单份用量，供库存出库扣减与消耗预测使用。
- **菜品标签体系：** 标签在 V1.0 基础上扩展，支持自定义标签，用于推荐引擎的内容特征提取。

<a id="33-订单管理"></a>
## 3.3 商户管理端——订单管理

- **列表与筛选：** 展示订单号、来源（桌号/外卖/自取）、菜品摘要、金额、支付状态、出餐状态、下单时间；按状态筛选、按订单号或桌号搜索。
- **订单详情：** 查看完整菜品明细（含备注）、金额构成、优惠使用、支付与状态流转记录。
- **状态流转：** 支持出餐中→已完成、待支付→已支付/已取消等状态更新；退款为敏感操作需权限与确认。
- **多源订单汇聚：** 订单可来源于用户端下单、点餐智能体生成、管理端代客录入或群体点餐会话，统一汇入订单流水。

<a id="34-桌台管理"></a>
## 3.4 商户管理端——桌台管理

- **桌台平面：** 以卡片网格展示台位编号、座位数与状态（空闲/就餐中/已预订/待清理），就餐中显示当前消费额与已用时长，已预订显示预订人与时间。
- **二维码生成：** 每个桌台自动生成专属二维码（含商户ID+桌台ID），顾客扫码直达该商户该桌台的点餐页。
- **状态操作：** 支持开台、并台/换台、结账、标记清理完成等操作。

<a id="35-营销活动"></a>
## 3.5 商户管理端——营销活动

- **优惠券：** 创建与维护满减、新客立减、会员专享等券；展示已领取、已核销与核销率。
- **套餐与促销：** 维护套餐（原价/现价）、组合优惠与限时促销（如时段特价、买一送一）；展示已售与状态。
- **智能推荐联动：** 进行中的活动自动同步至点餐推荐智能体，智能体在推荐时优先考虑活动商品并在推荐理由中提及优惠信息。
- **效果跟踪：** 对各活动的领取/核销/销量进行统计，数据供经营分析智能体评估营销 ROI。

<a id="36-库存管理"></a>
## 3.6 商户管理端——库存管理

分为食材库存、出入库记录、供应商三个子页。

- **食材库存：** 展示食材（名称、分类、当前库存、安全阈值、预测可用天数、状态：充足/临期/预警/紧张），低于安全阈值的食材高亮预警并推送至运营总览。
- **出入库记录：** 记录每笔入库、出库、损耗的时间、类型、食材、数量、操作人与备注。
- **供应商：** 维护供应商档案（名称、主营品类、联系人、电话、合作状态、历史履约评分），供采购建议匹配。

<a id="37-会员管理"></a>
## 3.7 商户管理端——会员管理

- **会员概览：** 展示会员总数、活跃数、本月新增、复购率、会员等级分布等指标。
- **会员列表：** 展示会员（姓名、等级：黑卡/金卡/银卡/普通、积分、到店次数、累计消费、最近到店、流失风险标记），按等级筛选、按姓名/手机号搜索。
- **会员档案：** 查看单个会员的基本信息、等级、积分与储值、历史订单、口味偏好标签、忌口信息、活跃度趋势。
- **流失预警：** 系统自动标记长期未到店（如超过 30 天）的高价值会员，支持定向推送优惠券挽回。
- **等级与积分：** 商户自定义等级规则与积分抵扣比例。

<a id="38-员工管理与排班"></a>
## 3.8 商户管理端——员工管理与排班

- **员工列表：** 展示员工（姓名、角色、在岗状态、本周工时），支持添加与编辑。
- **账号与角色：** 为员工分配登录账号与角色，角色决定商户管理端可访问的模块与操作权限（角色权限体系基于 RBAC）。
- **排班：** 以周排班表维护早/晚/休班次，统计工时；经营分析智能体可输出排班建议供参考。

<a id="39-系统设置"></a>
## 3.9 商户管理端——系统设置

- **商户信息：** 维护餐厅名称、Logo、地址、营业时间、联系电话、公告等，同步至用户端展示。
- **角色权限：** 配置本商户各角色对模块与操作的访问权限。
- **智能体参数：** 对各智能体的关键行为参数提供配置入口（详见 5.5），包括：
    - 点餐推荐：每次推荐数量、推荐策略权重（协同过滤/内容/热销/利润的占比）、是否开启群体点餐、是否开启营养建议。
    - 库存预测：安全库存系数、预测周期、是否自动生成采购单。
    - 客服处理：自动回复置信度阈值、强情绪自动转人工开关、是否允许自动发送回复。
    - 经营分析：摘要生成频率、异常检测灵敏度、What-if 模拟开关。

<a id="4-用户端功能需求"></a>
# 4. 用户端功能需求

用户端为面向顾客的微信小程序，定位为「**多商户餐厅发现 + 智能点餐 + 在线服务**」入口。V2.0 核心变化：新增商户选择与发现、点餐推荐大幅加强（群体点餐、营养建议、多模态输入）。

<a id="40-商户选择与首页"></a>
## 4.0 商户选择与首页

- **附近商户：** 顾客打开小程序后，基于微信 LBS 展示附近已开通的商户列表（以距离排序），每家商户展示 Logo、名称、品类标签、距离、评分、月销。
- **商户搜索与筛选：** 支持按名称搜索、按品类（中式快餐/面馆/茶餐厅/火锅/日料等）筛选。
- **最近访问：** 记录顾客最近访问的商户，快速进入。
- **收藏商户：** 支持收藏常去的商户，收藏列表优先展示。
- **进入商户：** 点击商户卡片进入该商户的菜单与点餐页。也可通过扫描桌台二维码直接进入指定商户指定桌台。

<a id="41-扫码进入与菜单浏览"></a>
## 4.1 扫码进入与菜单浏览

传统菜单浏览与手动加购通道保留，与 4.2 节的对话式智能点餐**并行存在**，顾客可自由选择习惯的点餐方式。

- **扫码直达：** 顾客扫描桌台二维码进入用户端，自动绑定商户与桌号（URL 参数含 merchant_id + table_id），默认展示菜单浏览页，同时底部常驻「AI 点餐助手」入口。
- **菜单浏览：** 按分类浏览菜品，展示图片、名称、价格、标签（招牌/热销/特价/新品/限时/健康轻食/高蛋白/素食/人气Top10）、销量与评分。**支持直接点击菜品选择规格、数量、备注并加入购物车。**
- **智能排序：** 菜品默认排序由推荐引擎生成的个性化排序（结合顾客历史偏好与全局热销），而非固定排序。
- **菜品详情：** 展示主料、详细描述、过敏原标注、营养信息（热量/蛋白质/碳水/脂肪）、价格、用户评价摘要；支持选择数量与备注，**一键加入购物车**。
- **库存可见性：** 已停售或缺货菜品置灰不可下单。
- **AI 点餐助手入口：** 页面底部常驻悬浮按钮「🤖 AI 点餐」，点击切换至对话式智能点餐界面（4.2 节）。从对话界面也可切回菜单浏览，购物车数据双端同步。

<a id="42-对话式智能点餐核心"></a>
## 4.2 对话式智能点餐（核心——推荐点餐入口）

用户端提供对话点餐入口，接入**深度升级的「点餐推荐智能体」**（详见 5.1）。这是整个项目的**核心交互模块**，与 4.1 节的传统菜单浏览**并行存在**。顾客可自由选择：习惯自助浏览的顾客走传统菜单加购流程，需要推荐帮助的顾客使用 AI 对话点餐。

> **设计原则：** 传统菜单通道与 AI 对话通道并存，但 AI 对话是**推荐的首选入口**。原因：(1) 智能体相比顾客自己浏览菜单能发现更符合偏好的菜品组合；(2) 智能体自动处理忌口/过敏原/营养约束，人工浏览容易遗漏；(3) 群体点餐、智能凑单等高级功能仅在对话流中可实现；(4) 每一次对话交互都是推荐引擎的学习信号，使其越用越准。两个通道的购物车数据实时同步，顾客可在任意时刻切换通道。

### 4.2.1 多模态输入

- **文本输入：** 自然语言表达点餐需求（如「来份招牌牛肉面，不要香菜，再配杯解腻的冷饮」）。
- **语音输入：** 支持微信语音转文字，顾客可直接说出需求，由智能体识别与处理。
- **图片识别（可选高级功能）：** 顾客拍摄菜品图片或菜单照片，智能体识别并推荐相似菜品。

### 4.2.2 意图理解与多轮对话

- **深层意图理解：** 不只识别「点单/修改/询问」基础意图，还能理解：
    - **模糊需求澄清：** 「来点清淡的」→ 主动追问「您是想吃汤面类还是凉拌类？是否需要低脂低卡的选择？」
    - **场景识别：** 「今天带了客户来吃饭」→ 识别为商务宴请场景，推荐高价高评招牌菜。
    - **情绪感知：** 「随便来点吧，赶时间」→ 识别急切情绪，推荐出餐快的菜品。
    - **人数感知：** 「我们三个人」→ 切换为群体点餐模式。
- **多轮上下文维护：** 整个对话维护完整上下文字段（已点列表、忌口、预算、偏好、场景），不丢失信息。

### 4.2.3 混合推荐展示

- **推荐卡片：** 智能体输出的推荐以富卡片形式展示，包含菜品图片、名称、价格、推荐理由（如「根据您的口味偏好」「今日特价」「与已点的牛肉面经典搭配」「高蛋白低脂，符合您的健身需求」）。
- **推荐多样性：** 单次推荐 2–4 个菜品，覆盖不同类别（主菜+配菜+饮品），避免同质化。
- **一键加入：** 推荐卡片上的菜品一键加入购物车。
- **实时购物车摘要：** 对话中始终显示当前订单摘要（菜品列表+合计金额），一目了然。

### 4.2.4 群体点餐模式

- **发起群体点餐：** 顾客说「我们五个人一起吃」或在桌台绑定多人后自动进入群体模式。
- **个人偏好融合：** 智能体获取同席各顾客（如通过微信授权）的偏好，综合推荐兼顾各人口味。
- **冲突解决：** 当口味冲突时（如有人不吃辣有人无辣不欢），智能体建议折中方案或分点菜品。
- **预算分摊：** 支持群体 AA 制预估，智能体在推荐时考虑人均预算。
- **协同确认：** 群体点餐结果可分享至群友确认后统一下单。

### 4.2.5 营养与健康管理

- **营养分析：** 当前订单的营养成分汇总（总热量/蛋白质/碳水/脂肪），以可视化方式展示。
- **健康建议：** 基于顾客的健康目标（减脂/增肌/均衡）给出调整建议（如「当前总热量偏高，是否将炒饭替换为蒸饭？」）。
- **过敏原严格检测：** 智能体在推荐菜品前自动检查顾客忌口与过敏原，有风险时明确警告并排除。

### 4.2.6 推荐反馈闭环

- **推荐反馈：** 顾客可对推荐进行「不感兴趣」「已点过」「太贵了」等快速反馈，智能体即时调整后续推荐。
- **推荐解释：** 顾客可点击「为什么推荐这个？」获得透明解释。

<a id="43-购物车与下单支付"></a>
## 4.3 购物车与下单支付

- **购物车：** 查看与修改已选菜品、数量与备注，实时计算金额与可用优惠。
- **智能凑单：** 当购物车金额接近优惠门槛时（如「再凑 ¥8 可减 ¥5」），智能体推荐凑单菜品。
- **优惠使用：** 选择可用优惠券/套餐，自动校验门槛并抵扣。
- **下单与支付：** 提交订单并通过微信支付完成在线支付。
- **支付确认：** 支付成功后生成订单并进入出餐流程。

<a id="44-订单跟踪"></a>
## 4.4 订单跟踪

- **订单状态：** 查看当前订单的出餐进度（已下单/出餐中/已完成）与明细。
- **催单：** 支持一键催单，通知触发后厨/管理端。
- **历史订单：** 登录会员可查看历史订单（跨商户），支持再来一单（快速复购，自动进入对应商户）。

<a id="45-在线客服"></a>
## 4.5 在线客服

- **智能应答：** 基于升级的客服智能体（详见 5.3）进行多轮客服对话。
- **投诉与建议：** 顾客可提交投诉或建议，智能体识别类别与情绪、给出初步回应。
- **转人工：** 对强情绪投诉或复杂问题，顾客可转人工，对应工单进入对应商户管理端。
- **主动服务：** 智能体在订单延迟等场景下主动推送通知。

<a id="46-会员中心"></a>
## 4.6 会员中心

- **注册/登录：** 支持微信授权登录或手机号+验证码注册/登录。
- **多商户会员：** 一个账号可在多家商户分别成为会员，积分与等级按商户独立管理。
- **积分与等级：** 查看在各商户的积分、等级与权益。
- **优惠券：** 查看与领取各商户发放的优惠券。
- **个人偏好管理：** 维护口味偏好标签（喜辣/清淡/偏爱牛肉/海鲜过敏等）、健康目标（减脂/增肌/无特殊需求）、忌口信息。这些数据跨商户共享（即在不同商户点餐时同一偏好生效）。
- **收藏：** 收藏的商户与菜品。

<a id="47-评价反馈"></a>
## 4.7 评价反馈

- **订单评价：** 顾客对已完成订单进行多维评分（口味/服务/环境/性价比）与文字评价，支持上传图片。
- **菜品级评价：** 可针对单个菜品单独评价。
- **评价用途：** 评价数据进入经营分析智能体进行情感分析与差评归因，同时用于推荐引擎的内容特征更新（高评分菜品提升推荐权重）。

<a id="48-在线预订"></a>
## 4.8 在线预订

- **桌台预订：** 顾客选择商户、日期、时段与人数发起预订。
- **预订确认与提醒：** 商户确认预订，临近时间向顾客推送提醒。

<a id="5-ai-智能体与-dify-工作流详细需求核心"></a>
# 5. AI 智能体与 Dify 工作流详细需求（核心）

本章为项目重点。四个智能体均基于 Dify 实现，在 V1.0 基础上进行了**全面的复杂度升级与功能扩展**。对话类场景采用 Chatflow，批处理场景采用 Workflow，复杂推理场景引入 Agent 节点与多步工具调用。**点餐推荐智能体（5.1）是本章的重中之重。**

<a id="50-智能体总体设计"></a>
## 5.0 智能体总体设计

### 5.0.0 智能体必要性设计（AI-Native 设计原则）

本系统的核心设计理念是「**AI 原生（AI-Native）**」——智能体不是附加功能，而是业务流程的不可替代组件。以下从四个维度论证每个智能体的必要性：

| **智能体** | **为什么不用智能体会显著变差** | **如果用传统方式替代** | **必要性类型** |
|---|---|---|---|
| **点餐推荐** | 智能体提供千人千面的个性化推荐、忌口/过敏原自动检测、智能搭配与凑单——这些能力传统菜单浏览完全不具备。同时智能体是群体点餐、营养管理等高级功能的唯一入口。 | 顾客手动翻菜单逐项挑选，无个性化推荐、无搭配建议、忌口靠肉眼检查——「能用但远不如智能体好用」。群体点餐等高级功能完全无法使用。 | **体验依赖（Experience Dependency）** |
| **库存预测** | 食材 50+ 种 × 多因子（趋势/周期/节假日/天气/促销）× BOM 关联计算的复杂度已超出人工在合理时间内完成的能力。 | 采购靠经验拍脑袋 → 要么缺货要么积压损耗。人工做同等精度的多因子预测需数小时/天。 | **能力依赖（Capability Dependency）** |
| **客服处理** | 高峰时段同时涌入数十条咨询，1–2 名人工客服不可能即时响应每位顾客。且智能体同时提供流失预警、情绪追踪等人工无法实时执行的分析。 | 顾客等待超时 → 差评/流失。高峰期客服崩溃。人工无法同时追踪情绪趋势和流失风险。 | **规模依赖（Scale Dependency）** |
| **经营分析** | 跨维度（订单×评价×库存×客流×营销）的关联分析、异常根因自动下钻、What-if 量化模拟——这些分析的维度组合与计算量超出人脑处理边界。 | 管理者凭直觉决策 → 看不到隐性经营问题（如某菜品利润低但消耗高、差评集中在某时段）。 | **洞察依赖（Insight Dependency）** |

**智能体必要性总结：** 点餐推荐智能体虽与传统菜单通道并存，但其提供的个性化推荐、安全检测与高级功能（群体点餐/营养管理/智能凑单）是传统菜单无法比拟的——传统通道保证兜底可用性，智能体通道提供差异化核心竞争力。库存预测、客服处理、经营分析三个智能体在实践中不可替代（能力/规模/洞察依赖），分别解决「算不过来/响应不及/看不出来」的刚性痛点。

**教师验收关注点：** 答辩时评委可能会问「顾客不用 AI 对话，直接手动加购不也行吗」，回答策略——确实可以，但就像有了导航 app 之后很少人再翻纸质地图。AI 对话的价值在于：(1) 新顾客面对几十道菜无从下手时 AI 主动引导；(2) 有忌口/过敏原时 AI 自动把关（手动点餐容易遗漏）；(3) 多人就餐时 AI 协调众口难调。传统通道存在是为了不强迫用户改变习惯，但智能体通道显然是更好的选择——我们通过双通道数据对比（采纳率、客单价、满意度）来证明这一点。

### 5.0.1 Dify 集成架构

- **调用方式：** Dify 自托管于本地虚拟机，后端服务作为唯一调用方，经内网通过 Dify API 调用智能体，使用应用密钥鉴权。用户端不直接调用 Dify。
- **商户上下文注入：** 调用时通过输入变量传入 `merchant_id` 及该商户的业务上下文（菜单、库存、订单、会员、FAQ 等），由后端从业务库按 `merchant_id` 组装。
- **工具回写：** 工作流中通过自定义工具节点（HTTP 请求业务服务层接口）读取实时数据或回写结果。所有工具接口均校验 `merchant_id` 归属。
- **知识库（RAG）：** 菜单说明、门店 FAQ、退换货政策等按商户维度组织为知识库，供点餐与客服检索增强。
- **会话管理：** Chatflow 类智能体通过 Dify 会话 API 维护多轮对话上下文。

### 5.0.2 公共数据交互

| **交互对象** | **说明** |
|---|---|
| 读取上下文 | 商户菜单与库存状态、菜品 BOM、历史订单与销量（按时段/日期聚合）、会员偏好与忌口、评价数据、FAQ 知识库、营销活动、供应商信息、营业时段。 |
| 回写结果 | 订单/购物车变更、推荐日志（用于后续模型训练反馈）、采购建议与采购单、客服工单与回复草稿、经营摘要与异常告警、智能体运行日志。 |
| 运行日志 | 每次智能体动作记录 `merchant_id`、类型、输入摘要、输出摘要、耗时、采纳情况与结果，供「智能体动态」展示与效果评估。 |

### 5.0.3 智能体间协同

V2.0 新增智能体间的协同关系：

- **点餐推荐 ↔ 库存预测：** 实时库存状态影响推荐结果（缺货不推）。点餐消耗数据为库存预测提供实时输入。
- **点餐推荐 ↔ 经营分析：** 推荐采纳率与客单价变化反馈至经营分析，用于评估推荐策略效果。
- **客服处理 ↔ 经营分析：** 投诉归因结果汇总至经营分析，辅助发现运营问题。
- **库存预测 ↔ 经营分析：** 采购成本与库存周转率纳入经营 KPI。

### 5.0.4 输出形式与动作确认

智能体的输出根据动作类型分为两类：

**自动执行（无需人工确认）：**
- 点餐推荐：智能体生成的推荐直接展示给顾客，顾客通过对话反馈「要这个/不要」即为确认。推荐本身无需商户审核。
- 库存预警：智能体检测到的库存预警直接推送至运营总览与库存管理页，无需人工审核预警是否成立。
- 客服自动应答：高置信度 FAQ 类咨询，智能体直接回复顾客，不经过人工。
- 经营异常检测：智能体检出的异常直接推送告警，无需人工审核异常是否真实。

**需人工确认的关键动作：**
- 确认下单：由顾客在对话界面点击确认（智能体不能代顾客下单）。
- 生成采购订单：由商户管理者确认采购建议后生成正式采购单。
- 发送客服回复（强情绪/涉及金额）：草稿需人工确认后发送。
- 经营建议执行：排班调整、菜单变更等建议仅供决策参考，执行需管理者操作。

<a id="51-点餐推荐智能体重磅"></a>
## 5.1 点餐推荐智能体（重磅——体验依赖型智能体）

**业务目标：** 构建混合推荐引擎，以多轮对话理解顾客点餐意图，融合协同过滤、内容推荐、知识图谱、实时上下文与 LLM 语义理解，提供高度个性化、可解释、多样化的菜品推荐。支持群体点餐、营养健康管理、多模态输入与实时反馈适应，成为项目的核心技术亮点。传统菜单手动加购通道作为兜底方案并行存在，但智能体通道在个性化、安全性、高级功能上具有压倒性优势。

> ### ⚡ 为什么这个智能体不可或缺（答辩核心论点）
>
> **体验依赖（Experience Dependency）：传统菜单手动加购通道存在，但智能体在个性化推荐、安全检测、高级功能三个维度上具备传统方式无法企及的优势。**
>
> | **维度** | **传统菜单手动加购** | **AI 对话智能点餐** |
> |---|---|---|
> | **菜品发现** | 人工翻阅菜单（几十上百道菜），靠眼力筛选，容易错过感兴趣的菜 | 智能体理解需求后精准匹配 2–4 个候选，从海量菜品中秒级定位 |
> | **个性化** | 无——每个顾客看到同样的菜单 | 千人千面——基于历史偏好、口味、健康目标、场景动态调整 |
> | **忌口/过敏原保障** | 靠顾客自己阅读菜单小字标注，容易遗漏 → 有安全风险 | 智能体自动校验每道候选菜品的过敏原与顾客忌口，**零遗漏** |
> | **组合搭配** | 靠顾客自己判断「主食+配菜+饮品」是否协调 | 知识图谱驱动的搭配推荐，确保组合合理 |
> | **群体点餐** | 多人各自翻菜单、口头协商（耗时且冲突频发），**传统菜单无协调能力** | 智能体自动融合多人偏好、检测冲突、生成折中方案——**仅 AI 通道可用** |
> | **智能凑单** | 顾客自己心算优惠门槛，「差几块钱没享受到优惠」频发 | 智能体自动检测优惠门槛并推荐最优凑单菜品——**仅 AI 通道可用** |
> | **营养管理** | 不可能——顾客自己算每道菜的热量蛋白质 | 自动汇总订单营养成分 + 健康目标对比 + 替换建议——**仅 AI 通道可用** |
> | **冷启动** | 新顾客面对完整菜单无从下手 | 智能体通过对话快速探询偏好，首次体验即个性化 |
>
> **答辩场景预演：** 「顾客不用 AI 对话，直接手动加购不也行吗？」→ 确实可以——传统菜单通道作为兜底方案存在。但这就像有了手机导航之后仍然可以翻纸质地图——极少有人会选后者，因为 AI 通道的优势是碾压级的：个性化推荐让顾客发现意想不到的好搭配、过敏原自动检测是安全刚需、群体点餐/智能凑单/营养管理只有 AI 能做到。我们保留传统通道是尊重用户习惯，但产品价值由 AI 通道承载。

### 5.1.2 召回层详细设计

#### (A) 协同过滤召回（Collaborative Filtering Recall）

| 策略 | 说明 | 实现方式 |
|---|---|---|
| **用户-菜品协同过滤** | 基于「用户-菜品-评分/频次」矩阵，使用 SVD 或交替最小二乘法（ALS）降维，计算用户间/菜品间相似度。 | Dify 代码节点（Python）运行预训练的轻量模型。 |
| **同群用户偏好** | 按「口味画像+消费能力+用餐场景」对用户聚类，同一 cluster 内的高频/高评分菜品作为候选。 | 离线聚类 + 在线查询 cluster 候选集。 |
| **时段流行度** | 当前时段（如工作日晚市）该商户的热门菜品 Top N。 | 从业务库查询近期同时段销量排行。 |
| **会话内协同** | 同一桌台多人点餐时，利用已点菜品信息进行协同推荐。 | 从当前会话/购物车提取已点特征。 |

#### (B) 内容推荐召回（Content-Based Recall）

| 策略 | 说明 | 实现方式 |
|---|---|---|
| **菜品标签相似度** | 将菜品标签（分类/口味/食材/烹饪方式）向量化，计算与用户偏好向量的余弦相似度。 | TF-IDF / 轻量 Embedding + 向量检索。 |
| **口味向量匹配** | 将顾客的口味偏好（喜辣/清淡/偏爱牛肉等）编码为向量，与菜品口味向量匹配。 | Dify 代码节点计算。 |
| **营养目标匹配** | 减脂顾客优先推荐低卡高蛋白菜品，增肌顾客推荐高蛋白菜品。 | 基于菜品营养信息过滤排序。 |
| **场景匹配** | 识别用餐场景（独自用餐→单人份/单人套餐，商务宴请→招牌大菜+酒水，家庭聚餐→大份量+儿童友好），匹配对应菜品。 | LLM 节点做场景识别 + 规则过滤。 |

#### (C) 知识图谱 & 规则召回

| 策略 | 说明 | 实现方式 |
|---|---|---|
| **搭配知识图谱** | 构建「菜品—搭配—菜品」知识图谱（如红烧牛肉面←搭配→凉拌黄瓜、酸梅汤），经典搭配作为推荐候选项。 | 知识图谱存储（Neo4j 或 JSON 规则库），代码节点查询。 |
| **口味相容规则** | 确保推荐的菜品组合口味协调（如不推荐麻辣火锅+冰激凌同时作为主推荐，但可作为场景化提醒）。 | 规则引擎。 |
| **营养搭配规则** | 一餐推荐中确保荤素搭配、主食+菜品平衡。 | 代码节点进行组合校验。 |
| **促销/库存规则** | 优先召回有促销活动或需要清库存的菜品。 | 从业务接口获取促销与临期数据。 |

### 5.1.3 精排层详细设计

| 排序因子 | 说明 | 权重 |
|---|---|---|
| **CTR 预估** | 基于菜品在相似上下文中的历史点击/采纳率，使用逻辑回归或轻量 GBDT 预估点击率。 | 商户可调（默认 0.3） |
| **用户偏好匹配度** | 召回阶段计算的相似度分数。 | 商户可调（默认 0.25） |
| **利润贡献度** | 菜品的毛利（售价 - 成本），商户可配置利润导向权重。 | 商户可调（默认 0.15） |
| **时效/新鲜度** | 新品/特价/限时标签加权；近期已点过的菜品降权（避免重复推荐）。 | 固定 |
| **多样性（MMR 重排）** | 使用 Maximal Marginal Relevance 算法对最终推荐列表进行多样性重排，确保推荐列表覆盖不同品类。 | 商户可调 |
| **LLM 语义相关性** | LLM 对「用户当前表达需求 vs 菜品描述」的语义匹配打分。 | 商户可调（默认 0.1） |

### 5.1.4 LLM 策略调控层

LLM 在推荐引擎中扮演「策略调控者」与「解释生成者」双重角色：

- **场景感知调控：** LLM 识别用餐场景后，动态调整各召回路的权重。例如「商务宴请」加大招牌菜与高价菜权重，「日常便餐」加大性价比与出餐速度权重。
- **推荐理由生成：** 为每个推荐菜品生成自然语言解释：「这道酸辣粉是本周热销 Top 3」「根据您的口味偏好——偏爱麻辣」「与您已点的红烧牛肉面是经典搭配」「高蛋白低脂，符合您的健身目标」。
- **安全校验：** 在最终输出前，LLM 再次校验推荐列表是否包含顾客忌口/过敏原菜品，如有风险则替换并说明。
- **拒绝/质疑处理：** 当顾客对推荐表示「不感兴趣/太贵了/吃过了」时，LLM 理解反馈并调整策略（降低相关特征权重，切换推荐方向）。

### 5.1.5 Dify Chatflow 节点设计（主工作流）

| **节点** | **类型** | **说明** |
|---|---|---|
| 开始 | 起始 | 接收用户消息、`merchant_id`、`table_id`、`member_id`、当前购物车等输入变量。 |
| 多模态预处理 | 条件分支+工具 | 判断输入类型（文本/语音/图片）。语音调用语音识别工具转文字；图片调用图像识别工具提取特征。 |
| 场景与意图识别 | LLM | 深层意图解析：基础意图（点单/修改/询问/确认）+ 用餐场景（日常便餐/商务宴请/家庭聚餐/朋友聚会/独自用餐）+ 紧急程度 + 人数感知 + 预算感知。输出结构化意图对象。 |
| 用户画像获取 | 工具(HTTP) | 调用业务接口获取会员偏好标签、忌口、健康目标、历史订单、近 30 日消费统计。无历史则使用冷启动策略。 |
| 协同过滤召回 | 代码(Python) | 基于用户画像与历史数据，运行协同过滤算法，输出 Top 15 候选菜品。 |
| 内容推荐召回 | 代码(Python) | 基于菜品标签向量与用户偏好向量相似度，输出 Top 15 候选菜品。 |
| 规则/图谱召回 | 代码+工具 | 查询搭配知识图谱获取已点菜品的搭配推荐；查询促销/库存接口获取活动菜品与临期菜品。输出 Top 10 候选。 |
| 多路融合去重 | 代码 | 将三路召回结果合并、去重，执行忌口/过敏原/缺货过滤，输出融合候选集（约 20–30 个）。 |
| 精排 | 代码 | 多因子加权排序 + MMR 多样性重排，输出 Top 5–8 候选菜品（含排序分数）。 |
| LLM 策略调控 | LLM | 接收精排候选列表，根据场景与对话上下文做最终策略调整，选择 2–4 个最终推荐，生成推荐理由与对话回复。 |
| 安全检查 | 条件分支 | 再次校验最终推荐是否含过敏原/忌口冲突。有冲突则回退到融合层换候选。 |
| 推荐日志记录 | 工具(HTTP) | 记录本次推荐详情（推荐项、召回来源、排序分数、上下文），用于离线效果评估与模型迭代。 |
| 回复与推荐输出 | 结束 | 输出对话回复文本 + 结构化推荐列表（`[{dish_id, name, price, reason, image_url, tags}]`）+ 购物车变更指令。 |

### 5.1.6 子工作流：群体点餐协调（GroupOrderFlow）

当识别为群体点餐场景时，调用此子工作流：

| **节点** | **类型** | **说明** |
|---|---|---|
| 成员偏好汇总 | 工具(HTTP) | 获取同席所有成员的偏好标签、忌口并集、健康目标。 |
| 冲突检测 | 代码 | 检测口味冲突（如 A 不吃辣、B 喜辣）、预算差异、忌口冲突。 |
| 折中方案生成 | LLM | 针对冲突生成折中推荐方案（如「辣度可选的菜品」「分点辣与不辣的菜搭配」）。 |
| 人均预算计算 | 代码 | 根据人数与总预算计算人均，约束推荐菜品价格区间。 |
| 群体推荐输出 | LLM | 综合各方偏好与约束，输出群体推荐列表与协调说明。 |

### 5.1.7 子工作流：营养健康顾问（NutritionAdvisorFlow）

当顾客开启营养模式或系统识别到健康目标时调用：

| **节点** | **类型** | **说明** |
|---|---|---|
| 订单营养汇总 | 代码 | 计算当前购物车所有菜品的总热量、蛋白质、碳水、脂肪。 |
| 目标对比 | 代码 | 与顾客健康目标（如减脂：< 600 kcal/餐）对比，输出偏差。 |
| 调整建议生成 | LLM | 生成具体的菜品替换/减量建议与理由（如「将炒饭换蒸饭可减少约 200 kcal」）。 |
| 膳食日志更新 | 工具(HTTP) | 下单后记录本餐营养摄入，累积生成顾客的膳食日志（可选）。 |

### 5.1.8 子工作流：推荐反馈学习（FeedbackLearningFlow）

顾客对推荐进行操作后触发（异步）：

| **节点** | **类型** | **说明** |
|---|---|---|
| 反馈收集 | 工具(HTTP) | 获取顾客对推荐的操作（采纳/忽略/负反馈/点击查看详情）。 |
| 特征更新 | 代码 | 根据反馈更新该用户-菜品交互记录；正反馈→增强关联，负反馈→降权对应特征。 |
| 冷启动衰减 | 代码 | 新用户随着交互增多，逐步从热销推荐过渡到个性化推荐。 |

### 5.1.9 冷启动策略

对于新顾客（无历史订单/偏好）或新商户（无历史数据），采用分层冷启动策略：

| **场景** | **策略** |
|---|---|
| 新顾客 + 成熟商户 | 以该商户的全局热销 + 当前时段流行 + 新客特价为主，对话中引导顾客表达偏好，快速积累画像。 |
| 老顾客 + 新商户 | 利用该顾客在其他商户的口味偏好作为迁移特征，结合新商户的菜品标签做内容推荐。 |
| 新顾客 + 新商户 | 以商户设定的推荐权重 + 分类多样性 + 通用搭配规则为主，配合 LLM 主动探询偏好。 |

### 5.1.10 推荐效果评估与 A/B 测试

- **在线指标（实时监控）：** 推荐采纳率、推荐点击率（CTR）、推荐带来的客单价提升、推荐覆盖的订单占比。
- **离线指标（定期评估）：** 精确率（Precision@K）、召回率（Recall@K）、NDCG@K、推荐多样性指数。
- **A/B 测试框架：** 支持商户配置多套推荐策略权重方案，对比不同方案的采纳率与客单价，数据驱动策略优化。

### 5.1.11 验收指标

| **指标** | **目标值** | **说明** |
|---|---|---|
| 推荐相关性 | ≥ 90% | 人工评估推荐合理且无缺货/停售/忌口冲突的比例。 |
| 推荐采纳率 | ≥ 35% | 被顾客采纳的推荐占总推荐次数的比例。 |
| 对话成功率 | ≥ 92% | 能正确完成「加项/去配料/改数量/下单」操作的比例。 |
| 群体点餐满意度 | ≥ 85% | 群体点餐场景下所有成员认为推荐兼顾了各自需求的比例。 |
| 冷启动采纳率 | ≥ 20% | 新顾客首次点餐的推荐采纳率。 |
| 推荐多样性 | ≥ 0.7 | 推荐列表中不同品类的覆盖率（1 - 同品类占比）。 |
| NDCG@5 | ≥ 0.75 | 推荐排序质量（基于采纳数据的离线评估）。 |

## 5.2 库存预测智能体（能力依赖型智能体）

**业务目标：** 基于多维度时序数据预测食材消耗与可用天数，结合供应商可靠性评分与动态安全库存策略，智能预警并生成可执行的采购计划，最小化缺货与损耗成本。

> ### ⚡ 为什么这个智能体不可替代（答辩核心论点）
>
> **能力依赖（Capability Dependency）：食材 50+ 种 × 多因子（趋势/周期/节假日/天气/促销）× BOM（每道菜关联 3-10 种食材）→ 计算复杂度已超出人工在合理时间内完成的能力边界。**
>
> | **维度** | **人工方式** | **本智能体** |
> |---|---|---|
> | **消耗预测** | 店长凭经验估计「牛肉明天大概用多少」 | 基于 90 日历史 × 周期因子 × 促销弹性 × 时段分布的数学模型 |
> | **计算规模** | 50 种食材，人脑只能直觉估计 5–10 种 | 50+ 种食材 × 未来 7 日的逐日预测，秒级完成 |
> | **预警及时性** | 发现库存见底才匆忙采购 | 提前 3–7 天预警，预留采购周期 |
> | **采购量计算** | 「大概再买一箱吧」 | 精确计算：覆盖至下个采购周期 + 基于需求波动性的动态安全库存 |
> | **供应商选择** | 习惯性找老供应商 | 基于准时率/品质/价格/周期的多因子评分自动推荐最优供应商 |
> | **损耗预防** | 食材过期才发现、扔掉 | 提前标记临期食材 + 建议关联菜品促销消化 |
>
> **答辩场景预演：** 「店长自己看库存不行吗？」→ 50 种食材 × 考虑未来一周 × 周末/天气/促销波动 = 350+ 个预测值，即使有经验的店长也只能凭直觉覆盖少数核心食材。且 BOM 关联意味着一个菜品销量变化会连锁影响 5–8 种食材——这种多维传导效应人脑无法精确追踪。

### 5.2.1 V2.0 升级要点

相比 V1.0 的简单移动平均，V2.0 升级为：
1. **多因子时序预测：** 趋势 + 周期（周循环）+ 节假日效应 + 天气 + 促销因子。
2. **供应商评分体系：** 基于历史履约（准时率/质量/价格波动）为供应商打分，采购建议自动优选。
3. **动态安全库存：** 安全库存不再固定，而是基于需求波动性（标准差）动态计算。
4. **损耗优化建议：** 临期食材的消化方案（推荐关联菜品促销或套餐搭配）。
5. **跨食材替代建议：** 某种食材短缺时，推荐可替代食材（基于 BOM 与口味相似度）。

### 5.2.2 触发与输入

- **触发：** 定时（每日开市前/收市后）自动运行，或在管理端「库存预测」工作台手动触发。
- **输入：** `merchant_id`、历史销量（按菜品/食材×日/时段）、当前库存与安全阈值、菜品 BOM、临期信息、供应商与采购周期、节假日日历、促销计划。

### 5.2.3 Dify Workflow 节点设计

| **节点** | **类型** | **说明** |
|---|---|---|
| 开始 | 起始 | 接收触发信号、`merchant_id` 与时间范围参数。 |

<a id="53-客服处理智能体"></a>
## 5.3 客服处理智能体（规模依赖型智能体）

**业务目标：** 实现从「被动应答」到「主动服务」的升级。多轮对话理解顾客深层诉求，情绪变化实时追踪，流失风险提前预警，并在订单异常时主动触达顾客。

> ### ⚡ 为什么这个智能体不可替代（答辩核心论点）
>
> **规模依赖（Scale Dependency）：高峰时段并发会话数远超人力上限，且智能体同时执行「应答+情绪追踪+流失预警+主动触达」四种人类无法并行的任务。**
>
> | **维度** | **人工方式** | **本智能体** |
> |---|---|---|
> | **并发处理** | 1 人同时只能处理 1–2 个会话 | 同时处理全部会话，无上限（受限于服务器算力） |
> | **响应速度** | 忙时顾客排队等待数分钟 | 每个会话 2–5 秒首次响应 |
> | **情绪追踪** | 依赖客服个人共情能力，无法量化记录 | 逐条消息实时情绪评分，生成情绪变化曲线 |
> | **流失预警** | 无法实时计算（需要调数据、做分析） | 实时计算 RFM 流失风险分，高风险顾客自动标记 |
> | **主动服务** | 不可能——人工客服不会主动发现订单异常并联系顾客 | 自动扫描异常订单 → 生成致歉通知 → 推送触达 |
> | **知识沉淀** | 客服离职带走经验 | 优质回复自动提炼入知识库，知识资产化 |
> | **多语言** | 需额外招聘多语种客服 | 自动识别语言并以对应语言回复 |
>
> **答辩场景预演：** 「雇两个客服不就行了？」→ 午餐高峰 12:00–13:00 同时涌入 20+ 条咨询，两名客服即使不间断工作，每人也要处理 10+ 个会话，顾客等待时间 5–10 分钟——在餐饮场景这不可接受。且客服无法同时执行流失预警计算和主动异常扫描。智能体不是「替代人工客服」，而是「做到人工客服根本做不到的事」。

### 5.3.1 V2.0 升级要点

1. **多轮对话状态追踪：** 不只单轮分类，而是追踪整段对话的状态变化与目标达成。
2. **情绪变化曲线：** 不只单条消息的情绪识别，而是分析情绪的上升/缓解趋势。
3. **主动服务触发：** 监测订单状态异常（出餐超时、支付失败），主动向顾客推送通知与致歉，而非等顾客来问。
4. **流失风险预警：** 分析会员的到店频率变化、投诉历史、消费下滑趋势，标记高风险流失顾客并推荐挽回策略。
5. **知识库自增长：** 客服人工解决的优质回复，经审核后自动提炼为 FAQ 条目入库。
6. **多语言支持：** 识别顾客语言（中文/英文/方言特征），以对应语言回复。

### 5.3.2 Dify Chatflow 节点设计

| **节点** | **类型** | **说明** |
|---|---|---|
| 开始 | 起始 | 接收顾客消息、`merchant_id`、订单/会员上下文、会话历史。 |
| 语言检测 | LLM | 识别消息语言与是否含方言特征，决定回复语言。 |
| 深层意图分类 | LLM | 多标签分类：咨询/投诉/建议/预订/催单/退款请求/闲聊；同时识别子类别（如投诉→上菜延迟/口味/分量/服务态度/卫生）。 |
| 情绪评估 | LLM | 评估当前消息情绪（弱/中/强，正面/中性/负面）以及会话级情绪趋势（平稳/恶化/缓解）。 |
| FAQ 多路检索 | 知识检索(RAG)×2 | 并行检索商户 FAQ 知识库与通用餐饮知识库，融合排序后取 Top 3。 |
| 订单/会员上下文查询 | 工具(HTTP) | 如涉及订单，查询订单状态与详情；如为注册会员，查询会员等级与历史。 |
| 归因分析 | LLM | 投诉类深度归因：定位问题环节（前厅/后厨/供应链/系统），输出归因标签。 |
| 流失风险评估 | 代码+LLM | 基于会员到店频率趋势、投诉次数、消费下滑、最近到店间隔，计算流失风险分数。高风险顾客在工单中标记。 |
| 话术草拟 | LLM | 生成回复草稿：得体、共情、包含解决方案或补偿建议。高风险顾客追加挽回话术（如专属优惠券）。 |
| 分流判定 | 条件分支 | 规则+LLM 置信度综合判定：(a) 高置信+低情绪→自动回复；(b) 中情绪/涉及金额→草稿待人工确认；(c) 强情绪/安全相关→强制转人工+紧急标记。 |
| 工单回写 | 工具(HTTP) | 创建/更新客服工单。 |
| 主动服务触发判断 | 代码 | 检查是否有主动服务触发条件（订单超时未出餐、支付失败待处理），满足条件则创建主动触达任务。 |
| 结束 | 结束 | 输出分类、情绪、归因、回复草稿、流失风险、处理建议。 |

### 5.3.3 子工作流：主动服务引擎（ProactiveServiceFlow）

定时运行或事件触发，不等顾客来找：

| **节点** | **类型** | **说明** |

<a id="54-经营分析智能体"></a>
## 5.4 经营分析智能体（洞察依赖型智能体）

**业务目标：** 从「看数据」升级为「用数据决策」。自动生成多维经营摘要，检测异常并追溯根因，支持 What-if 场景模拟，输出可执行优化建议，并支持商户间的匿名化对标分析。

> ### ⚡ 为什么这个智能体不可替代（答辩核心论点）
>
> **洞察依赖（Insight Dependency）：跨维度（订单×评价×库存×客流×营销×会员）的关联分析、异常根因自动下钻、What-if 量化模拟——这些分析的维度组合与计算量超出人脑处理边界。**
>
> | **维度** | **人工方式** | **本智能体** |
> |---|---|---|
> | **异常检测** | 管理者发现营收骤降后才去翻数据找原因 | 实时监控多指标，自动检测 + 自动下钻（是哪个菜品/哪个时段/哪个客群出了问题） |
> | **跨维度关联** | 人脑最多关联 2–3 个维度（如「下雨天生意差」） | 同时分析营业额×菜品×时段×客流×评价×营销 ROI 的交叉影响 |
> | **菜单工程** | 凭感觉判断「哪个菜好卖哪个菜不好卖」 | BCG 矩阵精确分类（明星/金牛/问题/瘦狗），量化利润×销量 |
> | **What-if 模拟** | 拍脑袋估计「涨价会怎样」 | 基于历史弹性系数的量化模拟 + 敏感性分析 |
> | **RFM 客群** | 最多区分「常客」和「偶尔来的」 | R/F/M 三维度精细分群 × 各群消费贡献 × 流失风险 |
> | **自然语言查数** | 打开 Excel 手动拉表、做图 | 直接问「上周晚市哪个菜毛利最高」即可获得答案+图表 |
> | **对标分析** | 无——单店无法知道同行水平 | 匿名化品类聚合数据对比，了解竞争力位置 |
>
> **答辩场景预演：** 「管理者自己看报表不就行了？」→ 一个典型的中式快餐店每天产生 50–100 笔订单 × 每笔 3–5 道菜 × 7 天 = 1000–3500 条明细。加上评价情感、库存变动、客流时段、营销 ROI……即使全部做成 Excel 报表，管理者也无力逐项分析交叉关联。智能体不是「替你看报表」，而是「发现你看不到的隐性模式和异常」——就像体检 AI 发现医生肉眼看不到的早期病灶。

### 5.4.1 V2.0 升级要点

1. **异常检测与根因分析：** 不只报告「营收下降 15%」，而是自动下钻分析是哪类菜品、哪个时段、哪个客群导致的变化。
2. **What-if 模拟：** 「如果客单价提升 10%，月利润能增加多少？」「如果增加 2 张桌台，翻台率需要达到多少才回本？」
3. **菜单工程分析：** 基于 BCG 矩阵思想，将菜品分为「明星（高利润高销量）/金牛（高利润低销量）/问题（低利润高销量）/瘦狗（低利润低销量）」，给出菜单优化建议。
4. **顾客生命周期分析：** 按 RFM 模型（最近消费/消费频率/消费金额）细分客群，识别高价值顾客与流失风险顾客。
5. **竞品对标（可选）：** 结合该商户所在品类的匿名化聚合数据，给出竞争力参考。
6. **排班优化建议：** 基于客流时段预测，建议最优排班方案。

### 5.4.2 经营分析包含三条工作流

#### (A) 自然语言查数（Chatflow）

#### (C) What-if 模拟（Chatflow，按需触发）

| **节点** | **类型** | **说明** |
|---|---|---|
| 开始 | 起始 | 接收模拟假设（如「客单价提高 10%」「增加晚市套餐」）。 |
| 假设参数解析 | LLM | 将自然语言假设解析为可计算的参数变更。 |
| 模拟计算 | 代码 | 基于历史数据与参数变更进行模拟计算（如客单价×1.1 × 历史订单量，扣除预估流失）。 |
| 结果解读 | LLM | 将模拟结果转化为商业解读与风险评估。 |
| 结束 | 结束 | 输出模拟结果、敏感性分析与建议。 |

### 5.4.3 验收指标

| **指标** | **目标值** | **说明** |
|---|---|---|
| 查数正确率 | ≥ 90% | 常见经营问题的查询结果与人工核对一致的比例。 |
| 异常检测召回率 | ≥ 85% | 真实经营异常被智能体检测到的比例。 |
| 摘要可用性 | ≥ 80% | 经营摘要被评估为「有参考价值」的比例。 |
| What-if 偏差 | ≤ 20% | 模拟结果与后续实际数据的偏差（回溯验证）。 |

<a id="55-智能体运行监控与配置"></a>
## 5.5 智能体运行监控与配置

- **运行监控面板：** 在运营总览展示各智能体的运行动态与「智能体动态」实时信息流；在「AI 智能体」分组中提供每个智能体的详细运行统计（调用次数、平均耗时、成功率、采纳率、异常告警）。
- **参数配置：** 每个智能体的关键参数在商户管理端「系统设置→智能体参数」中可配置（详见 3.9）。
- **日志与审计：** 保存每次智能体调用的输入摘要、输出摘要、耗时、采纳情况。日志按 `merchant_id` 隔离。提供日志检索与导出功能。
- **A/B 实验管理（点餐推荐专用）：** 支持为点餐推荐智能体创建多组参数方案，分流至不同顾客进行 A/B 测试，实时对比各组采纳率与客单价。商户可查看实验报告并一键应用优胜方案。
- **费用统计（如适用）：** 若大模型调用有成本，支持按商户统计智能体调用量与大模型 token 消耗。

<a id="62-关键实体关系"></a>
## 6.2 关键实体关系

- **商户（Merchant）** 是核心隔离实体，几乎所有业务表通过 `merchant_id` 外键关联。
- **会员（Member）** 平台级唯一，通过 **MemberMerchant** 关联表与各商户建立多对多关系（同一顾客在不同商户有独立的等级/积分）。
- **订单（Order）** 1—N 订单明细（OrderItem）N—1 菜品（Dish）；订单 N—1 桌台、N—1 会员、N—1 商户。
- **菜品（Dish）** N—N 食材（Ingredient），通过 BOM（DishIngredient）关联用量；食材 1—N 出入库记录、N—1 供应商。
- **推荐日志（RecommendationLog）** N—1 会员、N—1 商户、N—1 会话，独立于订单实体，用于推荐效果分析。
- **菜品搭配（DishPairing）** 自关联 Dish 表，形成菜品间的搭配网络。

<a id="63-数据采集与-ai-数据准备"></a>
## 6.3 数据采集与 AI 数据准备

- **销量数据：** 由订单明细聚合得到按菜品×食材、按日×时段的销量矩阵。
- **会员行为数据：** 订单历史、浏览/点击（如有埋点）、评价、偏好标签，构成用户画像。
- **知识库语料：** 菜单说明、门店 FAQ、政策、菜品搭配知识等按商户整理为 Dify 知识库。
- **评价情感：** 评价文本经情感分析得到情感标签与归因，沉淀为结构化结果。
- **推荐训练数据：** 推荐日志（RecommendationLog）中的「召回→推荐→采纳/忽略」数据，用于协同过滤模型的定期更新与 CTR 模型的训练。
- **演示数据：** 使用历史/模拟数据初始化各商户的销量、库存、评价等，使智能体在演示环境可正常产出。

<a id="64-多商户数据隔离策略"></a>
## 6.4 多商户数据隔离策略

| **层面** | **策略** |
|---|---|
| 数据库 | 所有核心业务表包含 `merchant_id` 列；业务查询在 ORM 层（SQLAlchemy）通过 context 自动注入租户条件；定时任务/智能体调用必须显式传入 `merchant_id`。 |
| 缓存 | Redis key 格式：`{env}:{merchant_id}:{entity}:{id}`，确保各商户缓存空间隔离。 |
| 文件 | 图片/文件存储路径：`/uploads/{merchant_id}/{entity}/{filename}`。 |
| API | 后端从 JWT Token 解析 `merchant_id`（商户端）或 `platform_admin_id`（平台端）；API 网关层校验请求的 `merchant_id` 与 Token 中一致。 |
| Dify 知识库 | 为每个商户在 Dify 中创建独立知识库/数据集；RAG 检索时限定在该商户范围内。 |

<a id="8-非功能需求"></a>
# 8. 非功能需求

<a id="81-性能"></a>
## 8.1 性能

- 常规页面交互响应 ≤ 1 秒；列表查询在中小数据量下流畅分页。
- 智能体响应：点餐推荐首次响应宜在 3–8 秒内返回（含多路召回与精排），必要时提供流式输出与「正在为您精选菜品…」的过渡反馈。客服智能体首次响应宜在 2–5 秒内。
- 批处理类（库存预测、经营摘要、主动服务扫描）以定时离线方式运行，不阻塞在线操作。
- 支持多商户并发运营（平台预估 10 个商户 × 每个 20 台位同时点餐 + 客服会话）。
- 数据库查询在 `merchant_id` 索引加持下保持高效。

<a id="82-安全与权限"></a>
## 8.2 安全与权限

- **多层级鉴权：** 平台端 JWT → `platform_admin_id`；商户端 JWT → `merchant_id + staff_id + role`；用户端 JWT → `member_id`（平台级会员）。
- **租户隔离刚性约束：** 后端 ORM 层自动注入 `merchant_id` 条件，数据库账号不共享，防止跨商户数据泄露。安全测试需覆盖「商户 A 的管理员尝试访问商户 B 的数据」等攻击场景。
- **敏感操作审计：** 退款、权限变更、商户审核、采购确认等操作记录完整的操作日志。
- **密钥保护：** Dify 密钥、微信支付密钥、短信密钥等仅由后端保管。
- **用户隐私：** 用户手机号、消费记录等按需脱敏展示；数据传输加密（HTTPS）。

<a id="83-可用性与易用性"></a>
## 8.3 可用性与易用性

- 管理端采用清晰的信息架构与一致的视觉语言，平台端与商户端使用统一的 UI 组件库。
- 用户端面向普通顾客，操作路径短（从打开小程序到完成下单 ≤ 5 步核心操作），对话点餐自然流畅。
- 智能体输出附带理由与可编辑/可确认入口，避免「黑箱」造成的不可控。
- 点餐推荐的推荐卡片设计醒目、理由清晰、一键操作。

<a id="84-可靠性与容错"></a>
## 8.4 可靠性与容错

- 工具回写需保证幂等并支持失败重试（最多 3 次），避免重复扣减库存或重复创建工单。
- **Dify 高可用保障：** Dify 部署采用多实例 + 健康检查，确保智能体服务的可用性。库存预测、客服处理、经营分析三个智能体为刚性依赖——不提供手动替代方案。点餐推荐智能体在 Dify 不可用时，系统自动隐藏 AI 对话入口，顾客仍可通过传统菜单通道完成点餐（不影响核心下单流程），待 Dify 恢复后 AI 入口重新开放。
- 关键数据（订单、库存、支付回调）具备一致性保障与异常告警。
- 智能体调用超时（>15s）时自动重试一次，仍超时则向用户返回「服务繁忙，请稍后重试」提示。

<a id="85-可维护性与可扩展性"></a>
## 8.5 可维护性与可扩展性

- 前端组件化、后端模块化，业务与智能体解耦。
- Dify 工作流可视化编排，便于调整节点、提示词与知识库而不改动业务代码。
- 推荐引擎的召回/排序/策略层松耦合，各层可独立升级（如替换协同过滤算法或增加新的召回路）。
- 多商户架构天然支持水平扩展：新增商户只需创建数据空间，不触及系统架构。
- 数据层预留分库分表扩展空间（当商户数增长至百级以上时可按 `merchant_id` 分片）。

<a id="86-兼容性"></a>
## 8.6 兼容性

- 管理端兼容主流桌面浏览器最新版本；用户端兼容主流移动机型的微信/浏览器环境。
- 接口与数据格式遵循统一规范（JSON、统一错误码），便于多端复用。
